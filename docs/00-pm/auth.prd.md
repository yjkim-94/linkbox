# PRD: auth (회원가입 / 로그인)

> Product Requirements Document — linkbox
> Generated by PM Agent Team (Discovery + Strategy + Research + Execution)
> Date: 2026-05-25 | Project Level: Dynamic (Next.js + bkend.ai)

---

## Executive Summary

| 관점 | 내용 |
|------|------|
| **Problem** | 사용자가 저장한 북마크와 태그가 기기·세션에 종속되면 다른 기기에서 접근할 수 없고, 데이터 소유권이 불분명해 신뢰가 생기지 않는다. |
| **Solution** | 이메일/비밀번호 기반 계정(회원가입·로그인·로그아웃)과 JWT + Zustand 기반 세션 유지, 보호 라우트로 "내 데이터"를 안전하게 귀속시킨다. |
| **Function UX Effect** | 한 번 로그인하면 새로고침·재방문에도 세션이 유지되고, 비로그인 시 보호 페이지 접근이 차단되어 마찰 없이 개인 공간이 보장된다. |
| **Core Value** | "내 링크는 어디서나 내 것" — 모든 후속 기능(bookmark, search)의 데이터 귀속 기반이 되는 신뢰 레이어. |

auth는 linkbox의 **첫 번째이자 토대 기능**이다. bookmark/search가 의미를 가지려면 데이터가 특정 사용자에게 귀속되어야 하며, auth가 그 귀속의 단일 진입점이다. 본 PRD는 MVP 범위를 "이메일/비밀번호 인증 + 세션 유지 + 라우트 보호"로 한정하고, 소셜 로그인·비밀번호 재설정 등은 명시적으로 후순위(Out of Scope)로 분리한다.

## 1. Discovery — 기회 분석 (Teresa Torres OST)

### 1.1 5-Step Discovery Chain

**Step 1 — Brainstorm (해결 가능한 문제 발산)**
- 다른 기기에서 내 북마크에 접근 불가
- 내 데이터가 누구 것인지 불분명 (소유권/신뢰)
- 비로그인 상태에서 개인 페이지가 노출될 위험
- 로그인 마찰(재로그인 반복)로 이탈
- 계정 복구 불가 시 데이터 영구 손실 불안

**Step 2 — Assumptions (가정 식별)**
- A1: 사용자는 개인 북마크를 위해 계정 생성 의향이 있다
- A2: 이메일/비밀번호만으로 초기 진입 마찰이 충분히 낮다
- A3: JWT를 클라이언트에 보관해도 MVP 보안 수준에서 수용 가능하다
- A4: 세션 유지가 재방문 리텐션의 핵심 동인이다
- A5: bkend.ai Auth가 회원가입/로그인/토큰 발급을 안정적으로 제공한다

**Step 3 — Prioritize (Impact × Risk 우선순위)**

| 가정 | Impact | Risk | 우선순위 | 검증 필요성 |
|------|:------:|:----:|:--------:|------------|
| A5 (bkend.ai Auth 동작) | High | High | **P0** | 가장 먼저 기술 검증 |
| A3 (JWT 클라 보관 수용) | High | Med | **P1** | 보안 정책 결정 필요 |
| A4 (세션 유지 = 리텐션) | High | Med | **P1** | 동작으로 검증 |
| A2 (이메일/PW 마찰 낮음) | Med | Low | P2 | 완화 가능 |
| A1 (계정 생성 의향) | High | Low | P2 | 시장 가정 |

**Step 4 — Experiments (검증 실험)**
- E1 (A5): bkend.ai Auth signup/login API 스파이크 — 토큰 발급/검증 왕복 확인
- E2 (A3): 토큰 저장 위치 비교 (메모리+httpOnly 쿠키 vs localStorage) PoC
- E3 (A4): 세션 복원 시나리오 테스트 — 새로고침/탭 재오픈/만료 후 동작 확인

**Step 5 — Opportunity Solution Tree**

```
Outcome: 사용자가 "내 링크는 안전하게 내 것"이라 신뢰하고 재방문한다
│
├─ Opportunity 1: 데이터 소유권 확립 (가입 진입)
│   ├─ Solution: 이메일/비밀번호 회원가입
│   ├─ Solution: 가입 즉시 빈 개인 보드 제공 (소유감)
│   └─ Solution: 중복 이메일/약한 비밀번호 즉시 인라인 검증
│
├─ Opportunity 2: 마찰 없는 재진입 (세션 유지)
│   ├─ Solution: JWT + Zustand 세션 상태
│   ├─ Solution: 새로고침/재방문 시 자동 세션 복원
│   └─ Solution: 토큰 만료 시 부드러운 재인증 유도
│
└─ Opportunity 3: 안전한 경계 (라우트 보호)
    ├─ Solution: 비로그인 시 보호 라우트 → 로그인 리다이렉트
    ├─ Solution: 로그인 상태에서 /login 접근 시 보드로 리다이렉트
    └─ Solution: 로그아웃 시 토큰·상태 완전 제거
```

**Top 3 검증 기회**: Opportunity 1·2·3 모두 MVP 필수 — auth는 토대 기능이므로 세 기회가 모두 충족되어야 후속 기능이 동작한다.

## 4. Execution — ICP, Beachhead, GTM

### 4.1 ICP (Ideal Customer Profile)
다기기를 쓰며 레퍼런스 링크를 자주 다루는 **개인 지식근로자/개발자**. 빠른 가입과 신뢰 가능한 세션 유지를 중시하고, 데이터 소유권에 민감하다.

### 4.2 Beachhead Segment (4-기준 스코어링)

| 기준 | 평가 | 점수(1-5) |
|------|------|:--------:|
| 접근성 | 개발자 커뮤니티/직접 가입으로 도달 가능 | 4 |
| 지불의향(초기 무료) | 무료 검증 단계, 의향보다 사용 검증 우선 | 3 |
| 통점 강도 | 다기기 동기화 통점 강함 | 5 |
| 입소문 잠재력 | 개발자/리서처 네트워크 전파력 | 4 |

> **선택 비치헤드**: "다기기 사용 개발자·리서처". 통점이 강하고 전파력이 높아 초기 검증과 확산에 유리.

### 4.3 GTM 전략
- **채널**: 직접 웹 가입(MVP) → 개발자 커뮤니티/콘텐츠(확장)
- **핵심 메시지**: "로그인 한 번, 어디서나 내 링크"
- **활성화 지표**: 가입 → 첫 북마크 저장까지의 시간/비율
- **성장 루프(초기)**: 만족한 사용자가 링크 공유/추천 → 신규 가입 (auth는 이 루프의 진입 게이트)

### 4.4 Battlecard 요약
- vs 브라우저 북마크: "기기에 묶이지 않는 진짜 내 공간"
- vs 무거운 올인원: "북마크에만 집중한 가벼운 전용 도구, 30초 가입"

## 5. PRD 본문 (8 Sections)

### 5.1 Overview / 목적
linkbox 사용자에게 개인 데이터 귀속의 기반이 되는 인증 기능을 제공한다. 회원가입·로그인·로그아웃과 세션 유지, 보호 라우트를 MVP 범위로 구현한다.

### 5.2 Goals & Non-Goals
**Goals**
- 이메일/비밀번호 회원가입·로그인·로그아웃
- JWT + Zustand 기반 인증 상태 유지 및 세션 복원
- 보호 라우트 접근 제어

**Non-Goals (Out of Scope, 후순위)**
- 소셜 로그인(OAuth)
- 비밀번호 재설정/이메일 인증
- 다중 세션/디바이스 관리
- 역할/권한(RBAC)

### 5.3 User Stories (INVEST)

| ID | 스토리 | 수용 기준(요약) |
|----|--------|----------------|
| US-1 | 방문자로서 이메일/비밀번호로 가입하고 싶다 | 유효 입력 시 계정 생성·자동 로그인, 중복 이메일/약한 PW 인라인 에러 |
| US-2 | 사용자로서 이메일/비밀번호로 로그인하고 싶다 | 정확한 자격 증명 시 보드 진입, 오류 시 명확한 메시지 |
| US-3 | 사용자로서 로그아웃하고 싶다 | 토큰·상태 완전 제거, 보호 라우트 접근 차단 |
| US-4 | 사용자로서 새로고침/재방문에도 로그인이 유지되길 원한다 | 유효 토큰 존재 시 세션 자동 복원 |
| US-5 | 사용자로서 비로그인 시 내 데이터가 보호되길 원한다 | 보호 라우트 접근 시 /login으로 리다이렉트 |

### 5.4 기능 요구사항 (FR)
- FR-1: bkend.ai Auth로 signup/login API 연동, 성공 시 JWT 수신
- FR-2: 인증 상태를 Zustand `auth-store`에서 관리(user, token, status)
- FR-3: 앱 로드시 토큰 검증/세션 복원 로직
- FR-4: 보호 라우트 가드(미인증 → 로그인 리다이렉트, 인증 상태에서 /login → 보드 리다이렉트)
- FR-5: 로그아웃 시 토큰·스토어 초기화
- FR-6: 폼 검증(이메일 형식, 비밀번호 최소 길이) 및 서버 에러 표면화

### 5.5 비기능 요구사항 (NFR)
- 보안: 검증된 BaaS Auth 사용, 토큰 보관 정책 결정(권장: 가능 시 httpOnly 쿠키, 불가 시 메모리+제한적 스토리지)
- 성능: 로그인/세션 복원 체감 지연 최소화
- 접근성: 폼 레이블·에러 메시지 스크린리더 대응
- 규약 준수: 파일 kebab-case, 컴포넌트 PascalCase, `type` 우선, `any` 금지

### 5.6 의존성 / 가정
- bkend.ai Auth 가용성(A5, P0 검증 대상)
- `NEXT_PUBLIC_BKEND_PROJECT_ID` 환경변수 설정

## 7. Test Scenarios (User Stories 기반)

- TS-1 (US-1): 유효 이메일/PW 가입 → 계정 생성·자동 로그인 / 중복 이메일 → 인라인 에러
